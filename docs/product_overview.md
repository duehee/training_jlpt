## 1. 제품 요약

| 항목 | 내용 |
|------|------|
| 프로젝트명 | JLPT 개인 학습 에이전트 |
| 한 줄 정의 | JLPT 학습자의 약점을 AI가 진단하고, 맞춤형 문법 학습 경로를 제공하는 서비스 |
| 최종 목표 | 포트폴리오 가치가 높은 AI 백엔드 프로젝트 + 향후 상품화 가능성 확보 |
| 개발 방식 | 1인 개발 + 역할 분리형 에이전트 협업 |
| 예상 기간 | 4~5개월 |

---

## 2. 출발점

- 개발자 본인의 **JLPT N2 불합격** 경험에서 출발했습니다.
- 핵심 출발 문장: **"책을 펴도 뭐부터 해야 할지 모르겠다"**
- 단순 챗봇이 아니라 **실무형 AI 백엔드 프로젝트**를 만드는 것이 중요합니다.

---

## 3. 타깃 사용자

| 구분 | 설명 |
|------|------|
| 주 타깃 | JLPT N5~N2 독학 학습자 |
| 핵심 페르소나 | 현재 수준과 다음 학습 포인트를 모르는 학습자 |
| 확장 타깃 | N2~N1 독해/청해 보강이 필요한 학습자 |

---

## 4. 핵심 페인포인트

| ID | 문제 | 대응 방향 |
|----|------|----------|
| **P1** | 뭐부터 공부해야 할지 모르겠다 | 진단 테스트 + 학습 경로 추천 |
| **P2** | 청해가 잘 늘지 않는다 | Phase 2의 TTS 기반 청해 훈련 |
| **P3** | 문법을 맥락 없이 외운다 | Retrieval 기반 문법 설명 |
| **P4** | 독해 공부가 비효율적이다 | Phase 3의 청크 기반 독해 보완 |
| **P5** | 학습이 여러 도구에 흩어진다 | 통합형 학습 흐름 제공 |
| **P6** | 왜 틀렸는지 설명받기 어렵다 | 오답 분석 + 약점 추적 |

> **에이전트 참고**: 기능 제안 시 반드시 P1~P6 중 어떤 문제를 해결하는지 명시할 것.

---

## 5. 차별화 포인트

- 일반적인 JLPT 챗봇이 아닙니다.
- **진단 → retrieval → 설명 → 분기형 학습 흐름**을 하나로 묶습니다.
- 실제 실패 경험에서 출발한 프로젝트라 면접용 스토리텔링이 강합니다.
- 정적인 문제집 소비가 아니라 **적응형 학습 경험**을 지향합니다.

---

## 6. 기능 채택 판단 기준

새로운 기능이나 제안은 아래 **3개 중 2개 이상**에 해당할 때만 우선 검토합니다.

| # | 기준 |
|---|------|
| 1 | **페인포인트 해결**: P1~P6 중 하나 이상을 해결하는가 |
| 2 | **포트폴리오 가치**: 기술적 차별화(RAG/에이전트/평가/인프라)에 기여하는가 |
| 3 | **Phase 적합성**: 현재 Phase 범위 안에 들어오는가 |

> **에이전트 참고**: 이 기준은 **범위 크리프(scope creep) 방지**의 핵심 장치입니다.
> 기능 제안 시 민석(PM)이 이 3가지로 판단합니다.
> Phase 범위는 `service_flows.md` 4장을 참조하세요.
