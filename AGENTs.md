# AGENTs.md — Common Agent Operating Rules

> 최근 업데이트 : 2026-04-20

역할, 도구, 프로젝트와 무관하게 모든 에이전트가 공유하는 공통 행동 규칙.
프로젝트 고유 도메인 지식은 CLAUDE.md를 참조.
역할 정체성과 페르소나는 prompts/{role}_prompt.md를 참조.

## [INFORM] 프로젝트 컨텍스트 읽는 순서
1. AGENTs.md (이 파일): 공통 행동 규칙 — HOW to act
2. CLAUDE.md: 프로젝트 도메인 지식 — WHAT this project is
3. prompts/{your_role}_prompt.md: 자신의 페르소나와 역할 경계 — WHO you are
4. {working_dir}/AGENTs.md (존재 시): 해당 디렉터리의 로컬 규칙

여러 파일에 정보가 동시에 등장하면 더 구체적인 파일이 이긴다.
디렉터리-로컬 > 프로젝트(CLAUDE.md) > 공통(이 파일).

## [CONSTRAIN] 파일 경계 (공통 원칙)
- 프로젝트 루트 외부 경로에 절대 접근 금지
- 시스템 경로(/etc/, /usr/ 등) 절대 건드리지 말 것
- 가상환경(.venv/, node_modules/ 등) 절대 수정 금지
- 환경 파일(.env) 읽기 또는 수정 금지
- 프로젝트별 쓰기/읽기 전용 경로는 CLAUDE.md에 정의됨

## [CONSTRAIN] 도구 사용 (공통 정책)
- 허용: 프로젝트 내부 파일 I/O, 테스트 러너, 빌드 도구,
  git (읽기 전용 작업)
- 차단: curl, wget, sudo, rm -rf, chmod, git push --force
- 프로젝트 고유 도구 명령(pytest, alembic 등)은 CLAUDE.md에 정의됨
- 근거: 외부 네트워크 호출과 시스템 변경은 사람의 명시적 결정이 필요

## [CONSTRAIN] 리소스 한도
- 작업당 최대 도구 호출: 100회
- 동일 파일 편집 한도: 5회 (3회 초과 시 접근 방식 재검토)
- 세션 타임아웃: 60분

## [CONSTRAIN] 승인 필요 (공통 게이트)
- 파일 삭제
- 의존성 추가 또는 제거
- 기존 테스트 삭제 또는 스킵
- 인프라 설정 변경 (Docker, CI 등)
- 자신의 역할 경계 외부 파일 수정
- AGENTs.md, CLAUDE.md, 또는 prompts/* 수정
- 프로젝트별 승인 게이트는 CLAUDE.md에 정의됨

## [VERIFY] 검증 원칙
- 작업 완료 선언 전 프로젝트 표준 검증을 실행
  (구체 명령은 CLAUDE.md)
- 새 기능에는 새 테스트가 필요 — 테스트 없는 코드는 미완성
- 단정문(assertion)은 *의도된 동작*을 반영해야 한다 (실제 출력이 아님)
- 실패 우회를 위해 mock이나 단정문을 수정하지 말 것
- `# type: ignore`, `eslint-disable` 같은 검사 우회 사용 금지

## [CORRECT] 실패가 발생할 때
- 같은 접근이 3회 실패 → 다른 전략 시도
- 총 5회 실패 → 중단하고 사람에게 보고
- 같은 에러 메시지 3회 → 즉시 에스컬레이션
- 테스트 실패 시 파일 전체 재작성 금지 — 구체 원인을 수정

## [CORRECT] 루프 감지
- 세션별 파일 편집 횟수 추적
- 같은 파일 5회 초과 편집 → 접근 방식 재검토
- 검증 통과 없이 총 30회 초과 편집 → 에스컬레이션
- 같은 에러 3회 이상 반복 → 즉시 에스컬레이션

## [CORRECT] 실패 로깅
- 모든 의미 있는 실패는 CLAUDE.md §Past Failures → Rules에
  예방 규칙 1건을 추가한다
- 형식: `YYYY-MM-DD — 무엇이 일어났나 → 대신 무엇을 할 것인가`
- 기존 규칙은 절대 삭제하지 않는다 — 프로젝트 메모리이다

## [CONSTRAIN] 범위 규율
- 범위 외 리팩토링 금지
- 발견한 이슈는 보고만 하고 조용히 수정하지 말 것
- 요구사항이 불명확하면 묻는다 — 추측하지 않는다

## [CONSTRAIN] 역할 경계와 핸드오프
- 각 에이전트는 prompts/{role}_prompt.md에 정의된 역할 내에서만 동작
- 역할 간 결정에는 명시적 핸드오프 노트가 필요
- 핸드오프 형식: "{name}님 확인 필요: {reason}"
- 핸드오프에 포함할 것: 컨텍스트 요약, 필요한 결정,
  지금까지 완료된 작업

## [INFORM] 보고 형식
작업 완료를 선언할 때 다음을 포함한다.
1. 변경된 파일 목록
2. 검증 결과 (테스트 / 린트 / 타입 검사)
3. 미해결 항목 (있는 경우)
4. 발견했지만 수정하지 않은 이슈
